# v6.4.5 Stage B — Live Endpoint Live Guard Rejection Runbook

本 runbook 用于在**本地 live endpoint 配置**下，验证 v6.4.4 Live Guard 默认拒绝交易。

**本阶段仍然不执行真实实盘下单。** 预期结果是信号被 Live Guard 拒绝并写入 journal，Binance 账户状态保持不变。

## 目标

| # | 验证项 |
|---|--------|
| 1 | `LIVE_TRADING_ENABLED=false` 时，本地受控 `POST /tradingview` 必须被拒绝 |
| 2 | journal：`status=live_guard_rejected`，`skip_reason=live_guard_trading_disabled` |
| 3 | 不进入 Binance 下单 |
| 4 | Runtime one-shot 可按当前设计处理；测试结束后 `locked=true` |
| 5 | BTCUSDT 最终无持仓、无条件单、无普通委托 |
| 6 | reconcile `error_count=0` |

## 前置条件

在开始前逐项确认：

### 1. 确认当前是真实 live endpoint（必须先过 preflight 门禁）

**在生成 payload 或发送 POST 之前**，必须先打开 Dashboard preflight 并逐项确认：

```http
GET /dashboard/api/live-canary/preflight?token=YOUR_DASHBOARD_TOKEN
```

或在 `/dashboard/runtime-control?token=...` 顶部 **Live Canary Preflight** 面板查看。

**必须同时满足：**

| 检查项 | 预期 |
|--------|------|
| `is_live` | `true` |
| `live_guard.guard_active` | `true` |
| `canary_ready` | `false` |
| `blocking_reasons` | **必须包含** `live_trading_disabled` |

若 `canary_ready=true` 或缺少 `live_trading_disabled`，**停止 Stage B**——说明 live 执行开关或 guard 配置不符合拒绝验证场景。

CLI 辅助确认（`.env` 中 `BINANCE_BASE_URL` 必须为 `https://fapi.binance.com`）：

```bash
python -c "from app.config import Settings; from app.tv_sandbox import binance_env_label; s=Settings(); print(s.binance_base_url, binance_env_label(s.binance_base_url))"
```

必须输出 `https://fapi.binance.com live`。

### 2. Live Guard 安全开关（保持拒绝态）

`.env` 中必须满足（**不要在本阶段修改 `.env` 文件本身**，仅人工核对当前运行配置）：

| 变量 | Stage B 要求 |
|------|----------------|
| `LIVE_TRADING_ENABLED` | `false` |
| `LIVE_CONFIRM_PHRASE` | 不配置或故意错误（与 `LIVE_EXPECTED_CONFIRM_PHRASE` 不匹配） |
| `LIVE_REJECT_TRADINGVIEW_BY_DEFAULT` | `true`（默认） |
| `ENABLE_TRADING` | 建议 `true`（用于验证 guard 而非 dry_run；仍不会下单） |

preflight 必须显示 `canary_ready=false`，且 `blocking_reasons` **必须包含** `live_trading_disabled`（通常还含 `live_confirm_phrase_invalid`）。缺少 `live_trading_disabled` 时不要继续。

### 3. TradingView Alert 必须暂停

- 关闭或暂停所有指向本 bot 的 TradingView Alert
- 本测试**只允许** localhost 受控 `POST`
- 不要使用 ngrok / 公网 webhook

### 4. Runtime 与账户初始状态

打开 `/dashboard/runtime-control?token=YOUR_DASHBOARD_TOKEN`，确认：

- BTCUSDT 无持仓
- BTCUSDT 无条件单、无普通委托
- preflight `reconcile_summary.error_count=0`（或先执行一次 Reconcile）

## 为什么需要 One-Shot Unlock

执行顺序为：**Runtime lock → Live Guard → TV sandbox → plan → 下单**。

若 Runtime `effective_locked=true` 且未启用 one-shot，信号会在 Runtime 层以 `runtime_locked` 拒绝，**不会**到达 Live Guard。

Stage B 要验证的是 `live_guard_trading_disabled`，因此测试前需通过 Dashboard **One-Shot Unlock**（TTL 建议 300s）使 `effective_locked=false`。

> 测试 payload 使用 `source=local_canary` 且 `signal_id` 不以 `TV-` 开头，属于**非 TV 信号**。one-shot **不会**被 `maybe_consume_one_shot_for_tv_signal` 自动消耗；测试结束后请手动 **Lock** 恢复 `locked=true` 并清除 one-shot。

## 测试步骤

### Step 0 — 启动服务（live 配置，本地 only）

```bash
python main.py
```

仅监听 localhost（默认 `127.0.0.1:8000`），不要暴露公网。

### Step 1 — One-Shot Unlock

在 `/dashboard/runtime-control`：

1. One-Shot Unlock，reason 例如 `v6.4.5 stage b live guard reject test`
2. TTL 300
3. 确认 `effective_locked=false`，`one_shot.enabled=true`，`one_shot.remaining=1`

### Step 2 — 生成本地受控 payload

从 Dashboard 或 Binance 读取当前 BTCUSDT mark/close 价格，例如 `97500.00`：

```bash
python scripts/build_local_tv_payload.py --close 97500.00
```

脚本行为：

- **环境保护（Stage B 默认）**：仅当 `BINANCE_BASE_URL=https://fapi.binance.com` 且 `LIVE_TRADING_ENABLED=false` 时才生成；否则拒绝并退出
- 从 `.env` 读取 `WEBHOOK_SECRET`（**不打印 secret**）
- 写入 `_v645_live_guard_reject_payload.json`（**不打印 payload 原文**）
- 仅打印 `signal_id`、`symbol`、`side`、`close`、`sl`、`tp` 摘要

若 endpoint 不是 live 或 `LIVE_TRADING_ENABLED=true`，脚本会报错退出——这与 preflight 门禁一致，避免在 demo 或错误配置下误生成 Stage B payload。

每次测试使用新的 `signal_id`（含时间戳），避免 duplicate。

### Step 3 — 本地受控 POST

**不要**在命令行 echo secret 或打印 payload 内容。使用文件引用：

```bash
curl -s -X POST "http://127.0.0.1:8000/tradingview" ^
  -H "Content-Type: application/json" ^
  --data-binary "@_v645_live_guard_reject_payload.json"
```

PowerShell 等价：

```powershell
curl.exe -s -X POST "http://127.0.0.1:8000/tradingview" `
  -H "Content-Type: application/json" `
  --data-binary "@_v645_live_guard_reject_payload.json"
```

预期 HTTP 200，信号已接收（后台异步执行）。

### Step 4 — 验证 journal

在 Dashboard Runtime Control 的 Journal 区域，或：

```http
GET /dashboard/api/executions?limit=5&token=YOUR_DASHBOARD_TOKEN
```

找到 Step 2 的 `signal_id`，确认：

| 字段 | 预期 |
|------|------|
| `status` | `live_guard_rejected` |
| `skip_reason` | `live_guard_trading_disabled` |
| `error_message` | 含 “LIVE_TRADING_ENABLED=false” 说明 |

**不得**出现 `protected`、`entry_not_filled`（含成交）、或 Binance 订单 ID。

### Step 5 — 验证 Binance 侧无变化

Dashboard preflight 或 Runtime Control 市场摘要：

- BTCUSDT positions: 0 / flat
- algo-orders: 0
- open-orders: 0

### Step 6 — 恢复 Runtime 锁定

Dashboard 点击 **Lock**（reason 例如 `v6.4.5 stage b cleanup manual lock`）。

确认：

- `locked=true`
- `effective_locked=true`
- `one_shot.enabled=false`

### Step 7 — Reconcile

Dashboard 执行 **Reconcile**，确认 `error_count=0`。

### Step 8 — 清理临时文件

```bash
del _v645_live_guard_reject_payload.json
```

**不要**将该文件提交到 git。

## 预期结果摘要

```
POST /tradingview (local_canary, one-shot active)
  → trader.execute()
  → validate_live_guard_before_plan()
  → live_guard_trading_disabled
  → journal: live_guard_rejected
  → 无 Binance 下单
  → 手动 Lock → locked=true
  → BTCUSDT flat, reconcile clean
```

## 安全要求

- **不打印** `WEBHOOK_SECRET`
- **不打印** payload 原文（脚本与 runbook 均遵守）
- **不提交** `.env`、`logs/`、`data/`、临时 JSON
- **不修改** Binance API key
- **不修改** TradingView webhook / Alert 配置（仅暂停 Alert）
- **不部署** VPS
- **无一键实盘下单按钮**

python -m unittest discover tests -v
python -m compileall main.py app tests scripts
python -c "from app.trader import Trader; print('import ok')"

Live Guard 单元测试：

```bash

```

## 回滚到 demo 的步骤

Stage B 完成后，将运行环境切回 demo（**手动操作 `.env`，不要由脚本自动修改**）：

1. 停止 bot 进程
2. 将 `BINANCE_BASE_URL` 改回 `https://demo-fapi.binance.com`
3. 确认 `LIVE_TRADING_ENABLED=false`（保持）
4. 重启 bot，确认 preflight `is_live=false`，`canary_ready=false`（`not_live_environment`）
5. Dashboard Lock，确认 demo 下 Runtime 正常
6. 删除 `_v645_live_guard_reject_payload.json`（若仍存在）
7. 恢复 TradingView Alert（仅在 demo soak 需要时）
